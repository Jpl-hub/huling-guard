# 开源基线与技术选型

## 1. 文档目的

本文档用于记录当前系统的开源基线、技术选型依据和不采用方案。本文档服务于后续开发、维护、复现和替换评估。

## 2. 选型原则

技术选型遵循以下原则：

- 实时链路优先保证稳定与可部署
- 离线链路允许使用更重的初始化能力
- 主干能力保持单一路线，不引入并行备选栈
- 选型结果必须能够进入现有训练链、运行时链和发布链

## 3. 当前采用的开源基线

### 3.1 姿态估计

当前姿态估计基线为 `RTMO`，来源于 `MMPose` 体系。

采用原因如下：

- 支持实时多人姿态估计
- 已有成熟工程实现
- 适合离线骨架抽取与运行时推理统一
- 与现有训练和发布链兼容

### 3.2 房间先验初始化

当前房间先验初始化采用：

- `Grounding DINO`
- `SAM 2`

采用方式如下：

- 仅在安装或环境变化后执行
- 仅输出固定房间区域先验
- 不进入实时主链

### 3.3 时序状态识别

当前时序识别采用自研 `Scene-Pose Temporal Net`。

该模型不是直接复用现成跌倒分类器，而是基于以下输入构建：

- 骨架序列
- 关键点置信度
- 运动学特征
- 场景关系特征

### 3.4 事件层

当前事件层采用自研 `EventEngine`，将时序状态流稳定为业务事件流。

### 3.5 YOLO / RT-DETR 对照口径

`YOLO`、`YOLO-pose` 与 `RT-DETR` 不被否定，但当前不作为主运行时主干。

本系统要解决的核心任务不是“画出人体框”，而是连续判断 `normal / near_fall / fall / recovery / prolonged_lying` 五类过程状态。单纯的人体框或单帧类别无法稳定区分“床上正常躺卧”“地面长时间卧倒”“失衡后恢复”和“真实跌倒”。

当前选择 `RTMO` 的原因是它直接输出 17 点姿态与关键点置信度，能够进入后续运动学特征、骨架质量评估、时序 `Transformer` 与事件引擎。`RTMO` 本身也属于一阶段实时多人姿态估计路线，工程上吸收了 YOLO 系列强调实时性的设计思想；因此当前路线不是绕开实时视觉检测，而是把检测能力推进到更适合跌倒过程建模的骨架层。

当前系统已经在可视化层绘制由可见关键点推导出的人体外接框。该外接框用于让上传复核和模拟监看更直观，不作为第二套检测模型，也不改变运行时判定依据。

后续科研化对照中，可以加入 `YOLO-pose + LSTM/Transformer`、`RT-DETR person box + pose/temporal`、`MMAction2 / ST-GCN` 等离线基线实验。它们只用于同数据、同窗口、同评估口径下的对照，不进入前端运行时切换按钮。

## 4. 不采用方案

### 4.1 不采用 OpenPose 主线

`OpenPose` 不作为当前系统主线。原因如下：

- 工程负担较重
- 与当前主链不一致
- 不利于统一训练与运行时语义

### 4.2 不采用纯规则主线

纯几何阈值规则法不作为当前主线。原因如下：

- 对场景变化敏感
- 难以处理高混淆动作
- 不利于统一训练、评估和应用级验证

### 4.3 不采用实时重型分割链

房间分割能力不进入实时链路。原因如下：

- 会显著增加运行时负担
- 会增加部署复杂度
- 对当前应用目标收益有限

### 4.4 不采用双路线切换

系统不保留第二套主干模型或第二套运行时。所有优化均在当前单一路线内完成。

## 5. 当前判断

当前主路线保持为：

`RTMO -> Scene-Pose Temporal Net -> EventEngine`

房间先验初始化保持为：

`Grounding DINO + SAM 2`

该路线与现有训练链、运行时链、历史归档链和批量验证链保持一致，仍为当前最合适的工程主线。
