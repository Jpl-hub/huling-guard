# 系统设计说明

## 1. 文档目的

本文档用于说明护龄智守系统的产品定位、功能边界、技术架构、数据流程、运行时设计和交付方式。本文档面向产品实施人员、算法工程人员、前后端开发人员、测试人员和后续维护人员。

## 2. 产品定义

护龄智守是一套面向单房间固定摄像头场景的安全值守系统。系统持续接收视频输入，对房间内人员状态进行时序判断，并输出风险提示、事件记录、会话摘要和历史归档。

系统不以单帧识别结果作为最终输出，而是以连续状态流、事件流和会话流作为主要交付对象。

## 3. 目标用户

- 家庭照护者
- 护理站值班人员
- 养老机构管理人员
- 系统维护人员

## 4. 功能范围

### 4.1 核心识别能力

系统当前输出以下五类状态：

- `normal`
- `near_fall`
- `fall`
- `recovery`
- `prolonged_lying`

在此基础上，系统进一步生成以下事件：

- 高风险失衡提示
- 跌倒事件
- 长时间卧倒事件
- 恢复事件

### 4.2 应用能力

系统在运行时提供以下能力：

- 监看画面展示
- 实时接入管理
- 上传视频分析
- 当前判断与建议动作展示
- 风险时间线展示
- 最近提醒展示
- 当前记录摘要
- 历史记录归档
- 历史记录筛选与回看
- 引擎透视模式下的状态分布与运行参数展示

### 4.3 工程能力

系统同时具备以下工程能力：

- 单视频推理
- 批量视频推理
- 发布包部署
- 误报复查
- 区间标签回灌
- 定向再训练

## 5. 设计原则

### 5.1 单一路线原则

系统只保留一条主技术路线，不引入备用模型栈，不提供双路线切换。

### 5.2 应用优先原则

首页面向照护和值守使用，优先回答当前状态、当前风险和最近提醒。技术构成和运行参数放在独立页签中，不干扰日常使用。

### 5.3 数据一致性原则

训练链和运行时链共享同一套坐标语义、特征语义和状态定义。发现语义不一致时优先修正数据链与推理链，不通过新增兼容分支规避问题。

### 5.4 可追溯原则

所有整轮实验应生成训练摘要、样本级对比、应用级对比、发布包校验结果和轮次完成检查。运行时服务应支持会话归档与历史回看。

### 5.5 状态定义统一原则

五类状态的业务含义、标注边界、复核口径和阈值校准原则统一以《状态定义手册》为准。训练链、运行时链和人工复核不得使用不同定义。

## 6. 技术架构

### 6.1 房间先验初始化

房间先验初始化在部署前执行，输出固定房间的区域先验。当前采用：

- `Grounding DINO`
- `SAM 2`

该模块仅用于离线初始化，不进入实时推理主链。

### 6.2 实时姿态感知

实时姿态感知模块采用 `RTMO`，负责逐帧输出：

- 17 点人体骨架
- 关键点置信度

### 6.3 时序状态识别

核心时序模型为 `Scene-Pose Temporal Net`。模型输入包括：

- 标准化骨架序列
- 关键点置信度
- 运动学特征
- 场景关系特征

模型输出五类状态概率。

#### 6.3.1 模型内部结构

当前 `Scene-Pose Temporal Net` 采用混合时序结构，而不是单一 LSTM 或纯规则法。内部结构依次为：

1. `Pose Encoder`：对每个关节的 `(x, y, confidence)` 做逐关节线性编码。
2. `Quality-Aware Spatial Summary`：使用关键点置信度对关节表示做质量加权汇聚。
3. `Feature Fusion`：将帧级表示与运动学特征、场景关系特征、帧级质量特征拼接后映射到统一隐空间。
4. `Residual Temporal Blocks`：使用残差时序卷积块提取局部动作变化。
5. `Transformer Encoder`：引入位置编码后建模更长时间依赖。
6. `Quality-Weighted Pooling`：在时间窗汇聚阶段使用帧级质量分数对低质量帧降权。
7. `Output Heads`：同时输出帧级辅助分类、时间窗级主分类和风险分数。

当前默认结构参数为：

- `num_joints = 17`
- `pose_dim = 3`
- `kinematic_dim = 8`
- `scene_dim = 8`
- `hidden_dim = 192`
- `num_heads = 6`
- `depth = 4`
- `num_classes = 5`

当前仓库内训练模板 `configs/train_scene_pose.yaml` 使用的模型口径为：

- `kinematic_feature_set = v2`
- `kinematic_dim = 12`
- `scene_dim = 8`
- `hidden_dim = 192`
- `num_heads = 6`
- `depth = 4`

### 6.4 事件引擎

`EventEngine` 根据时序状态流和阈值配置生成业务事件。该模块负责稳定化，不负责替代主模型。

### 6.5 应用层

应用层由以下部分组成：

- `FastAPI` 接口服务
- `SQLite` 会话归档
- `Vue 3` 运行时前端
- Docker 部署封装

## 7. 数据流程

### 7.1 训练数据流程

训练数据主线为：

1. 原始视频清单生成
2. `RTMO` 离线骨架抽取
3. 特征缓存生成
4. 时间窗构建与骨架时序增强
5. 模型训练
6. 样本级、事件级和应用级评估
7. 发布包打包
8. 批量视频验证

### 7.2 运行时数据流程

运行时主线为：

1. 视频输入
2. 姿态估计
3. 特征生成
4. 时序模型推理
5. 事件引擎输出
6. 前端展示
7. 会话归档

#### 7.2.1 当前运行参数

当前运行时默认参数为：

- `window_size = 64`
- `inference_stride = 4`

运行时同时输出以下输入质量指标，用于判断骨架质量是否下降：

- `pose_quality_score`
- `mean_keypoint_confidence`
- `visible_joint_ratio`

## 8. 界面设计

### 8.1 实时值守

实时值守页用于当前时刻的值守判断，展示以下内容：

- 监看画面
- 当前结论
- 建议动作
- 最近提醒
- 连续状态时间线
- 当前记录摘要

该页面默认进入 `值守视图`。值守视图优先呈现：

- 当前是否安全
- 是否需要过去
- 最近发生了什么

实时值守页当前已接入三种真实入口：

- `实时接入`：通过摄像头编号、RTSP 地址或容器内视频路径把连续画面送入运行时
- `模拟监看`：在没有真实摄像头时按实时节奏回放固定机位样例流
- `上传分析`：上传新视频后异步生成时间线、提醒记录和整段过程结论

在 `引擎透视` 模式下，同一页面额外展示：

- 当前状态分布
- 风险变化
- 骨架质量
- 运行参数
- 模型输出概率

### 8.2 历史回看

历史回看页用于追溯与复核，展示以下内容：

- 历史记录统计
- 记录筛选
- 过程回放
- 关键阶段
- 风险高点
- 最近提醒

### 8.3 系统信息

系统信息页用于部署、维护和交接，展示以下内容：

- 系统用途与输入方式
- 状态定义与判断链路
- 关键运行参数
- 误报收敛原则

### 8.4 窄屏适配

运行时前端已提供窄屏适配规则，保证在手机浏览器中仍可完成以下操作：

- 查看当前状态、最近提醒和处理建议
- 切换模拟监看源
- 上传视频并查看分析进度
- 打开实时接入面板并启动或停止输入
- 进入历史回看与系统信息页

- 系统用途
- 可识别状态
- 模块构成
- 当前运行参数
- 质量控制原则

## 10. 数据库设计

系统使用 `SQLite` 作为运行时归档数据库，负责保存：

- 会话元数据
- 状态统计
- 事件统计
- 报告路径
- 历史查询索引

该设计满足单机值守部署、现场复现和轻量留档需求。

## 11. 交付设计

### 11.1 训练环境职责

训练环境使用 GPU 服务器，仅承担以下职责：

- 训练与评估
- 骨架抽取与批量验证
- 导出运行时发布包

### 11.2 运行环境职责

最终运行环境以本机部署为准，标准交付物包括：

- `runtime-release/`
- `runtime-demo/`
- `runtime-data/`
- `docker-compose.runtime.yml`

系统支持本机启动和 Docker 启动。Docker 版本采用前后端一体化封装：

- 前端在镜像构建阶段完成 `Vue` 静态构建
- 运行时服务在 `/dashboard` 提供前端页面
- 归档目录通过卷挂载到容器外部

## 12. 当前约束

- 不引入双路线切换
- 不引入备用规则主线
- 不在实时链路中加入重型分割模型
- 不以窗口级分数单独决定主干升级
- 不以单次训练结果替代应用级验证

## 13. 后续工作重点

- 继续完善历史回看与事件回放
- 持续收紧发布包校验与整轮实验检查
- 保持本机 Docker 交付链稳定可复现
