# 模型与实验说明

## 1. 文档目的

本文档单独说明系统中的模型链路、训练口径、评估口径与当前保留主干。它与系统级文档分工如下：

- `README.md`：说明系统用途、能力边界、部署方式和仓库结构
- `项目设计说明.md`：说明完整方案、模块职责与运行时结构
- `模型与实验说明.md`：说明模型内部结构、训练数据流、评估规则和当前主干版本

## 2. 模型链路总览

当前系统只保留一条识别主线：

`RTMO -> Scene-Pose Temporal Net -> EventEngine`

各模块职责如下：

1. `RTMO`
   负责逐帧人体姿态估计，输出 `17` 个关键点和关键点置信度。该模块只负责把视频画面转成可用于时序建模的骨架序列，不直接给出最终风险结论。

2. `Scene-Pose Temporal Net`
   负责对连续时间窗进行状态识别。该模块融合骨架坐标、运动学特征和场景关系特征，输出五类状态的窗级判断，同时给出辅助帧级输出和风险输出。

3. `EventEngine`
   负责把连续状态流稳定成事件流。该模块不替代模型，只负责把概率流转换为可处置、可归档、可回看的事件对象，例如跌倒事件、长时间卧倒事件与恢复事件。

4. `Grounding DINO + SAM 2`
   只用于离线房间先验初始化，不进入实时主链。该支路用于生成床、沙发、地面活动区和风险边缘等房间先验掩码，以便把“床上正常躺卧”和“地面异常卧倒”区分开来。

## 3. Scene-Pose Temporal Net 内部结构

### 3.1 输入组成

`Scene-Pose Temporal Net` 的输入不是原始视频帧，而是长度固定的时间窗特征序列。每个时间步由三部分组成：

1. `pose features`
   由 `RTMO` 输出的关节坐标和关键点置信度组成，单个关节默认使用 `(x, y, confidence)` 三维表示。

2. `kinematic features`
   由骨架序列进一步计算得到，当前主实验使用 `kinematic_feature_set = v2`，维度为 `12`，用于表示速度、位移变化、身体中心高度变化、朝向变化等运动学信息。

3. `scene features`
   由房间先验与当前人物位置关系构成，当前默认维度为 `8`，用于表达人物与床、沙发、地面活动区、风险边缘之间的相对关系。

### 3.2 网络结构

当前实现为混合时序结构，而不是单纯的 LSTM 或纯规则模型。网络内部顺序如下：

1. `Pose Encoder`
   对每个关节的 `(x, y, confidence)` 做逐关节线性编码。

2. `Quality-Aware Spatial Summary`
   不直接对全部关节简单平均，而是使用关键点置信度对关节表示做质量加权汇聚，得到帧级骨架表示。

3. `Feature Fusion`
   将帧级骨架表示与运动学特征、场景关系特征、帧级质量特征拼接，再投影到统一隐空间。

4. `Residual Temporal Blocks`
   使用残差时序卷积块提取短时间局部动态模式。该部分主要负责抓住跌倒前后的局部速度突变、姿态快速变化和低位停留模式。

5. `Transformer Encoder`
   在残差时序卷积之后加入带位置编码的 `Transformer Encoder`，用于建模更长时间范围内的依赖关系，例如“失衡后恢复”“跌倒后持续卧倒”等需要更长上下文才能区分的过程。

6. `Quality-Weighted Pooling`
   在时间窗汇聚阶段使用帧级质量分数对低质量帧降权，减少遮挡帧、关键点掉点帧和低置信度帧对最终窗级判断的污染。

7. `Prediction Heads`
   当前代码中保留四个输出头或输出分支：
   - `frame_head`：帧级辅助分类输出
   - `quality_head`：帧级质量预测输出，用于学习哪些帧值得信任
   - `clip_head`：时间窗级主分类输出
   - `risk_head`：风险输出

### 3.3 默认结构参数

当前代码层默认参数如下：

- `num_joints = 17`
- `pose_dim = 3`
- `kinematic_dim = 8`
- `scene_dim = 8`
- `quality_dim = 3`
- `hidden_dim = 192`
- `num_heads = 6`
- `depth = 4`
- `num_classes = 5`

当前仓库内训练模板 `configs/train_scene_pose.yaml` 使用的模型口径为：

- `kinematic_feature_set = v2`
- `kinematic_dim = 12`
- `scene_dim = 8`
- `quality_dim = 3`
- `hidden_dim = 192`
- `num_heads = 6`
- `depth = 4`
- `quality_loss_weight = 0.15`

自动实验脚本 `scripts/run_public_corpus_training.py` 在公开数据联合训练中使用更大的实验口径：

- `kinematic_feature_set = v2`
- `kinematic_dim = 12`
- `scene_dim = 8`
- `quality_dim = 3`
- `hidden_dim = 256`
- `num_heads = 8`
- `depth = 6`

因此，模板配置用于本地轻量复现实验，公开数据主实验以自动实验脚本生成的配置为准。

### 3.4 输出状态

系统当前识别的五类状态为：

- `normal`
- `near_fall`
- `fall`
- `recovery`
- `prolonged_lying`

这些状态不会直接裸输出给最终页面，而是会进一步进入 `EventEngine` 做事件稳定化处理。

## 4. 运行时口径

当前运行时默认口径为：

- `window_size = 64`
- `inference_stride = 4`

同时，运行时额外输出三项骨架质量指标：

- `pose_quality_score`
- `mean_keypoint_confidence`
- `visible_joint_ratio`

这些质量指标有两个作用：

1. 在运行时帮助判断当前骨架输入是否退化
2. 在产品端为“为什么当前结论不够稳定”提供解释依据

系统原则是：低质量骨架不能被当作高置信异常依据，必要时应降低判断强度，而不是强行输出高风险结论。当前主模型已经把帧级质量特征与质量加权汇聚纳入前向计算，而不只是把质量指标停留在运行时展示层。

## 5. 训练数据流

### 5.1 当前公开数据来源

当前训练主线使用的公开视频语料主要包括：

- `CaucaFall`
- `UP_Fall`
- `UR Fall`

系统不直接把原始视频送入主模型训练，而是统一执行以下数据流：

1. 生成原始视频 manifest
2. 使用 `RTMO` 离线抽取骨架
3. 构建姿态、运动学、场景关系特征
4. 切分固定时间窗
5. 训练 `Scene-Pose Temporal Net`
6. 执行窗口级、样本级、应用级评估
7. 打包发布包并进入运行时验证

### 5.2 数据覆盖检查

训练前强制检查 `raw manifest -> pose manifest` 覆盖率。覆盖不完整时直接终止训练，不允许残缺样本静默进入主训练链。

### 5.3 当前增强策略

在缺少真实摄像头和新增实拍数据的阶段，主训练链默认启用骨架时序增强。当前已接入的增强包括：

- 时间窗抖动
- 短时遮挡
- 骨架坐标噪声
- 关键点置信度缺失模拟

这些增强的目标不是“伪造新场景”，而是补偿固定机位视频在轻度遮挡、关键点波动和短时丢帧条件下的鲁棒性。

### 5.4 当前损失设计

当前主训练链同时优化三个目标：

- `clip_loss`：时间窗级主分类损失
- `risk_loss`：风险输出损失
- `quality_loss`：帧级质量辅助损失

其中 `quality_loss` 使用帧级质量目标监督 `quality_head`，再由预测到的帧级质量分数参与时间汇聚。这样做的目的不是单纯多加一个 loss，而是让模型在遮挡帧、掉点帧和低置信度帧较多时，更主动地降低这些帧对最终窗级判断的影响。

### 5.5 复查区间与难负样本权重

区间标签回灌支持两类信息：

- `label`：用于覆盖某一时间区间的状态标签
- `sample_weight`：用于提高该区间在训练采样中的权重

仅把 ADL 误报区间从 `normal` 覆盖为 `normal` 不会改变训练信号，因为这些窗口原本就是正常类。针对这种情况，当前训练链支持把人工复查出的正常误报区间标记为 hard negative，通过 `sample_weight > 1.0` 提高采样概率，让模型在下一轮训练中更频繁看到这些容易误判成跌倒的正常活动片段。

这一机制仍属于同一条 `RTMO -> Scene-Pose Temporal Net -> EventEngine` 主线，不引入第二套模型，也不通过界面文案掩盖误判。

### 5.6 过渡区间自动挖掘

当前公开数据天然更容易提供 `normal / fall / prolonged_lying` 这类粗粒度标签，`near_fall` 和 `recovery` 的原生监督明显不足。为了解决这一问题，当前主线新增了“过渡区间自动挖掘”能力：

- 仅在 `fall` 与 `prolonged_lying` 这类正样本片段内执行
- 使用骨盆高度变化、下沉速度、躯干倾斜和骨架质量联合定位动作锚点
- 在动作锚点前自动切出 `near_fall` 区间
- 在动作最低点之后、存在明显抬升时自动切出 `recovery` 区间

该能力的目标不是凭空创造第二套标签体系，而是为同一条时序主干补足“过程标签”。这能让模型学习“失衡前兆”和“恢复起身”的连续变化，而不只是把片段粗暴压成 `fall` 或 `normal`。

当前对应脚本为：

- `scripts/build_transition_interval_labels.py`

当前对应实现为：

- `src/huling_guard/data/transition_mining.py`

当前首轮自动挖掘结果如下：

- 扫描样本总数：`2711`
- 可挖掘正样本：`1081`
- 新增过渡区间：`240`
- 其中 `near_fall = 194`
- 其中 `recovery = 46`

按数据源分布：

- `near_fall`：`CaucaFall 39`、`UP_Fall 126`、`UR Fall 29`
- `recovery`：`CaucaFall 1`、`UP_Fall 41`、`UR Fall 4`

这说明该机制已经能在多源公开数据里稳定补出过程标签，而不是只在单一数据集上产生孤立样本。

### 5.7 当前轮次结论

截至 `2026-04-11`，基于自动过渡区间挖掘的两轮实验均未达到保留标准：

- `public_plus_ur_v17_transition_v1_quality005_seed2026`
  - `sample_macro_f1 = 0.6132`
  - 对公开验证集总体可用，但 `near_fall` 仍没有形成有效监督，不能替代当前保留主干
- `public_plus_ur_v18_transition_v2_strict_quality005_seed2026`
  - 严格版过渡区间从 `240` 段收紧到 `155` 段
  - `sample_macro_f1 = 0.4944`
  - 相比 `v13` 的 `macro_f1_delta = -0.1720`
  - 当前轮判定为 `hold`

因此，自动挖掘过渡标签这一方向目前仍处于研究验证阶段，不能直接进入发布主干。当前线上与交付口径不得冒充 `v17` 或 `v18` 已经优于保留主干。

## 6. 评估口径

当前实验不会只看一个窗口级分数，而是采用三层评估：

### 6.1 窗口级评估

用于衡量模型在固定时间窗上的分类能力，主要指标包括：

- `macro_f1`
- `accuracy`
- 每类 `precision / recall / f1`

### 6.2 样本级评估

将多个窗口还原回视频样本，评估单段视频的最终分类表现。该层更接近“系统面对一段实际过程视频时给出的整体判断”。

### 6.3 应用级评估

基于真实运行时与归档结果，评估系统在正常场景与异常场景下的实际表现。当前已经纳入：

- 事件级精度与召回
- 正常场景下的误报事件总数
- 期望状态为 `normal` 时的主导状态漂移率
- 正常场景下是否出现不应触发的事件

系统当前已经把“正常场景误报”和“主导状态漂移”纳入主干晋级门槛，不会因为训练分数上升就自动替换现有主干。

## 7. 当前主干与版本演进原则

### 7.1 当前保留主干

当前运行时保留主干为 `v7`。

保留 `v7` 的原因不是“最新”，而是它在同口径对比下仍然是当前最稳的一版。后续多轮实验虽然在某些局部类别或窗口级指标上有改善，但在样本级和应用级口径下没有同时守住整体表现，因此没有替换 `v7`。

### 7.2 版本晋级规则

任一新实验版本要替换当前主干，至少需要同时满足以下原则：

1. 不明显削弱样本级整体表现
2. 不明显削弱 `recovery` 等关键类别
3. 不增加正常场景下的误报事件负担
4. 不增加期望为 `normal` 场景下的主导状态漂移

也就是说，系统当前不是“谁训练分数高就上线”，而是采用训练级、样本级、应用级联合保留规则。

### 7.3 近期实验结论

截至目前，主干后续几轮实验结论如下：

1. `v13`
   公开数据联合训练后，样本级指标仍然可用，但在真实 ADL 正常视频上的误报压制没有达到足够稳定的替换条件，因此未替换主干。

2. `v14`
   引入 ADL 复查区间后，仅执行“normal 覆盖为 normal”的回灌。这种做法没有改变有效训练信号，因此结果与 `v13` 基本一致，没有形成真实增益。

3. `v15`
   对 ADL 误报区间加入 `sample_weight = 4.0` 的 hard negative 采样强化。结果显示：
   - 训练级与样本级指标同步下降
   - 正常 ADL 视频误报上升
   - 主导状态漂移明显恶化

   因此该方向被明确判定为失败，不再按高权重继续推进。

4. `v16`
   将 hard negative 权重下调到 `sample_weight = 2.0` 后再次验证，结果仍然失败。当前结构化结论为：
   - `macro_f1` 相比 `v13` 下降 `0.0935`
   - 样本级 `macro_f1` 下降 `0.0098`
   - 正常 ADL 视频 `with_incidents_rate` 从 `0.3500` 上升到 `0.4250`
   - 正常 ADL 视频主导状态漂移率从 `0.0000` 上升到 `0.2250`
   - 保留判读结果为 `hold`，`0/9` 项通过

由此可以得出两个明确结论：

- 单纯放大 hard negative 采样权重，不足以解决当前误报问题
- 当前更关键的缺口不是“正常类看得不够多”，而是“`near_fall / recovery` 过渡监督不足”

因此，当前主线的下一个优化重点已经转向“过渡区间自动挖掘 + 过程标签补强”，而不是继续沿用更激进的 hard negative 加权。

## 8. 交付边界

### 8.1 训练环境职责

GPU 训练机只负责：

- 数据准备
- 模型训练
- 批量视频验证
- 导出运行时发布包

### 8.2 运行环境职责

最终交付不依赖训练机。当前标准交付形态为：

- `runtime-release/`
- `runtime-demo/`
- `runtime-data/`
- `docker-compose.runtime.yml`

运行环境负责：

- 加载发布包
- 提供 FastAPI 接口
- 提供 Vue 前端页面
- 维护 SQLite 会话归档

因此，训练机只是产出平台，不是最终宿主。最终系统应以本机或服务器上的 Docker 运行时部署为准。

## 9. 当前阶段判断

截至当前阶段，系统已经形成了完整的 AI 应用主干：

- 模型链路明确
- 训练数据流明确
- 评估口径分层明确
- 运行时与产品端已打通
- 发布包可独立交付

但也应明确：

- 当前仍缺少真实摄像头接入下的大规模长期验证
- 当前仍需继续压正常场景误报
- 当前仍需继续提升前端表现力与历史回看说服力

因此，当前更准确的定位是：

**一套已经完成主干闭环、可部署、可演示、可持续优化的智能值守系统原型。**
