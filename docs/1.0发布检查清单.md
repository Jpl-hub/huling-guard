# 1.0 发布检查清单

## 1. 发布目标

1.0 版本的目标不是把所有研究问题一次性做完，而是交付一套可运行、可部署、可复盘、可继续迭代的完整系统。

当前 1.0 版本必须满足三条底线：

- 前端展示的所有状态、提醒、时间线和回看都来自真实运行链路。
- 部署形态固定为本机或云端 Docker 运行时，不依赖训练服务器常驻。
- 模型主线只有一条，不引入第二套模型或备用切换方案。

## 2. 当前版本判断

截至 `2026-04-11`，系统层已经具备 1.0 发布基础，但模型层还没有完成最终定版。

当前判断如下：

- `实时接入 / 模拟监看 / 上传视频复核` 三条输入链路已经打通。
- `监视器总览 / 实时值守 / 历史回看 / 交接班 / 运行引擎` 五页已经进入 1.0 收口阶段。
- `v29` 已完成样本级评估，整体超过 `v21`，但 `near_fall` 回落，晋级结论为 `review`；进入应用级复核前不能直接替换当前保留发布包。

## 3. 发布前必须完成

### 3.1 模型基线

- 明确保留的运行时发布包版本。
- 补齐该版本的样本级、事件级和应用级结果说明。
- 说明当前版本可发布的依据，以及哪些类别仍需继续优化。

### 3.2 运行时系统

- `实时值守` 页能稳定展示实时接入、模拟监看和上传分析三种模式。
- 上传视频后，页面能在分析过程中持续显示真实进度和局部结果。
- `历史回看` 页能完成关键词搜索、日期筛选、状态筛选、回放、关键时刻跳转和过程复核。
- `交接班` 页能基于真实留档生成摘要、重点复核列表和状态分布。
- `运行引擎` 页只保留用户、部署方和维护人员真正需要的信息。

### 3.3 部署与交付

- ECS 版 Docker 部署可稳定启动。
- 本地 Docker 版可按照文档拉起。
- 发布包、示例数据、运行说明和接口文档齐全。

### 3.4 文档

- `README.md`：说明系统定位、输入方式、页面结构、模型路线、YOLO 对照口径和部署方式。
- `docs/项目设计说明.md`：说明产品边界、架构和运行流程。
- `docs/模型与实验说明.md`：说明模型结构、训练口径和当前轮次结论。
- 本文档：作为 1.0 发布前的最终检查表。

## 4. 当前阻塞项

当前最主要的阻塞项只有两条：

1. `v29` 样本级整体已超过 `v21`，但 `near_fall` 仍需应用级复核，不能直接替换当前保留发布包。
2. README、历史回看页、交接班页和运行引擎页需保持同一套 1.0 命名与模型边界口径。

## 5. 建议的 1.0 口径

如果近期对外发布 1.0，建议口径统一为：

- 这是一套面向单房间固定机位场景的安全值守系统。
- 当前已经完整支持实时接入、模拟监看和上传视频复核。
- 当前版本重点解决“连续判断、事件提醒、历史回看、上传复核、交接班摘要和系统留档”。
- 模型路线统一表述为 `RTMO` 姿态估计 + 自研时序 `Transformer` + `EventEngine`，不把系统描述成 YOLO 检测框应用。
- 模型仍会继续迭代，后续版本重点围绕 `v29` 做应用级复核并提升 `near_fall` 稳定性。
