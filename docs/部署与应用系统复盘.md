# 部署与应用系统复盘

## 1. 文档目的

本文记录护龄智守 1.0 从模型产物到可访问 Web 应用的部署经验。它不是宣传材料，而是后续重新拉起系统、复查问题和继续迭代时的工程记录。

本次系统最终形成的交付形态是：

- 后端：`FastAPI` 运行时服务，负责视频输入、推理状态、事件流、历史归档和静态前端托管。
- 前端：`Vue 3 + Vite` 构建出的单页应用，作为监视器总览、实时值守、历史回看、交接班和运行引擎界面。
- 数据：`SQLite` 与运行目录保存历史留档、上传视频、报告文件和前端可回放资源。
- 部署：使用 Docker Compose 在云服务器或本机启动，不依赖训练 GPU 服务器常驻。

## 2. 最终部署结构

仓库中的部署相关目录如下：

```text
huling_guard/
├── frontend/             # Vue 3 前端源码与 dist 构建产物
├── scripts/              # 数据准备、训练、评估、部署脚本
├── src/huling_guard/     # 后端、运行时、模型与事件引擎代码
├── runtime-release/      # 可发布的运行时模型与配置
├── runtime-demo/         # 模拟监看视频和样例结果
└── runtime-data/         # 运行时数据、SQLite 留档和上传文件
```

实际部署时，`frontend/dist` 必须先构建完成；Docker 运行时再由后端统一托管前端页面和接口服务。

## 3. 本地构建流程

### 3.1 构建前端

```bash
cd frontend
npm install
npm run build
cd ..
```

构建完成后应出现：

```text
frontend/dist/index.html
frontend/dist/assets/*
```

### 3.2 启动 Docker 运行时

```bash
docker compose -f docker-compose.runtime.yml up --build
```

默认访问：

- 控制台：`http://127.0.0.1:18014/dashboard`
- 健康检查：`http://127.0.0.1:18014/health`

### 3.3 基础校验

```bash
curl -fsS http://127.0.0.1:18014/health
curl -fsS "http://127.0.0.1:18014/archives?limit=300"
```

健康检查应返回 `status = ok`。`/archives?limit=300` 用于验证历史留档接口可以支撑批量复核和交接班读取。

## 4. ECS 部署流程

本次云端部署采用一台 Ubuntu 22.04 ECS。服务器上放置项目目录，例如：

```text
/opt/huling-guard/app
```

推荐流程如下。

### 4.1 首次准备服务器

```bash
apt update
apt install -y docker.io docker-compose-plugin rsync ffmpeg
systemctl enable --now docker
```

`ffmpeg` 很关键。上传视频分析、视频转码、关键帧或前端可播放资源处理都可能依赖它。如果容器内没有可用的视频处理工具，上传后容易出现分析卡住、无法生成可播放文件或回放失败。

### 4.2 同步代码与构建产物

本地先完成前端构建，再同步到服务器：

```bash
rsync -az --delete frontend/dist/ root@<ECS_IP>:/opt/huling-guard/app/frontend/dist/
rsync -az --delete frontend/src/ root@<ECS_IP>:/opt/huling-guard/app/frontend/src/
rsync -az --delete src/ root@<ECS_IP>:/opt/huling-guard/app/src/
rsync -az README.md docs/ root@<ECS_IP>:/opt/huling-guard/app/docs/
```

注意：不要把服务器密码、SSH 私钥、云厂商控制台截图、临时 token 写入仓库或文档。部署命令中的 `<ECS_IP>` 和账号信息应由部署者本地替换。

### 4.3 重启服务

```bash
cd /opt/huling-guard/app
rm -f package.json package-lock.json
docker compose -f docker-compose.runtime.yml restart huling-guard-runtime
```

这里删除根目录误同步的 `package.json / package-lock.json` 是一次实际踩坑：前端包文件应在 `frontend/` 内，不应出现在应用根目录，避免后续判断项目结构时产生混淆。

### 4.4 云端验证

```bash
curl -fsS http://<ECS_IP>:18014/health
curl -sS -i "http://<ECS_IP>:18014/archives?limit=300" | head -40
curl -fsS http://<ECS_IP>:18014/ | grep -Eo 'assets/[^" ]+' | head
```

检查点：

- `/health` 返回 `ok`。
- `/archives?limit=300` 返回 `200 OK`，不能出现 `422 Unprocessable Entity`。
- 首页 HTML 中引用的是最新 `frontend/dist/assets` 文件。
- 浏览器打开控制台后，不应出现前端资源 404。

## 5. 本次实际踩坑

### 5.1 前端页面可以访问，但视频不显示

常见原因：

- 前端拿到的是旧的 `dist` 资源，浏览器仍在加载旧 JS。
- 上传视频没有完成分析，后端未生成可播放资源。
- 容器缺少 `ffmpeg` 或视频文件路径不在容器可访问范围内。
- 前端播放的是原始视频，但浏览器不支持该编码。

处理经验：

- 先确认 `frontend/dist` 已重新构建并同步。
- 再看后端上传任务状态和日志。
- 最后确认容器内视频路径、编码和静态文件映射。

### 5.2 `/archives?limit=300` 返回 422

原因是后端接口对 `limit` 设置了 `le=200`，而交接班和批量清理需要更大的窗口。最终处理为：

- 后端 `/archives` 查询上限调整为 `500`。
- 前端 `runtimeApi.archives()` 的上限同步调整为 `500`。
- 部署后用 `curl ".../archives?limit=300"` 验证。

### 5.3 前端容易变成“说明页”而不是“工具”

早期页面中存在大量灰色解释句、过大的标题、重复状态描述和类似汇报材料的文案。最终收口原则是：

- 页面只保留状态、数据和动作。
- 技术说明只放在运行引擎页，不干扰实时值守。
- 首页和实时页不解释系统为什么这样设计，只回答当前状态、风险和操作。
- 删除“用于、展示、查看、这段过程”这类叙事性文字，改为短标签和数值。

### 5.4 训练服务器不能成为运行依赖

GPU 服务器适合训练、评估和打包模型，不适合作为系统 1.0 的常驻运行条件。最终运行时应做到：

- 模型发布包已经落在仓库或发布目录中。
- ECS 或本机 Docker 可以独立启动。
- GPU 服务器关闭后，Web 应用仍能访问、上传、回看和展示已有模型输出。

## 6. 应用系统建设经验

### 6.1 AI 应用不是前端套壳

这个系统的关键不是把模型结果显示出来，而是把连续视频流整理成可使用的业务对象：

```text
视频源 -> 姿态序列 -> 状态流 -> 事件流 -> 留档 -> 回看 -> 交接班
```

只有形成这条闭环，系统才从“模型 demo”变成“应用系统”。

### 6.2 上传视频不是简单文件上传

上传后应立刻进入一个可观测过程：

- 上传完成
- 后端开始逐帧处理
- 已处理帧数更新
- 局部状态和时间线逐步可见
- 分析完成后进入历史回看或监视器总览

如果上传后页面长时间没有状态变化，用户会认为系统卡死。上传分析必须有进度、状态和失败信息。

### 6.3 模拟监看要诚实表达

没有真实摄像头时，模拟监看是合理的工程模式，但不能伪装成已经完成真实部署。更稳妥的口径是：

- 当前支持模拟监看、上传复核和实时接入入口。
- 真实接入依赖摄像头提供 RTSP、本机设备编号或容器可访问视频路径。
- 封闭云摄像头如果拿不到视频流，不能直接接入。

### 6.4 参数不应暴露给普通用户

`window_size`、`stride`、模型版本、阈值、设备类型等信息对维护人员有用，但对值守人员是噪音。它们应放在运行引擎或维护视图，不应出现在实时值守主界面。

### 6.5 删除和清理是正式产品必须具备的能力

上传源、无效留档、普通无提醒记录都需要清理入口。否则系统很快变成无法维护的数据垃圾场。清理逻辑要注意：

- 分析中的上传视频不要删除。
- 已被历史留档引用的视频不要直接删除。
- 删除前给确认提示。
- 删除后刷新列表和统计。

## 7. 关闭或释放云资源前检查

如果准备释放 ECS 或 GPU 服务器，建议先完成：

- 代码已提交并推送到 GitHub。
- `README.md` 和 `docs/` 已同步。
- 模型发布包、配置文件和必要样例数据已在仓库或可恢复位置。
- 不再需要服务器上的 `runtime-data` 历史留档；如果需要，应先打包下载。
- 云服务器没有保存唯一副本。
- 仓库中没有密码、私钥、token 或云控制台敏感信息。

ECS 停止后，公网 IP 访问会中断。释放实例后，系统盘和公网 IP 是否保留取决于云厂商资源策略；如果仍需保留环境，不能执行释放，只能停止或创建快照。

## 8. 后续如果重新启动

重新启动项目时，优先按以下顺序恢复：

1. 拉取 GitHub 最新代码。
2. 构建 `frontend/dist`。
3. 准备 `runtime-release`、`runtime-demo` 和必要 `runtime-data`。
4. 启动 Docker Compose。
5. 验证 `/health`、`/archives`、上传视频分析和历史回看。
6. 再决定是否打开 GPU 服务器继续模型实验。

## 9. 最终经验

这次项目最大的经验是：模型能力、运行时工程、前端体验和部署文档必须一起收口。只训练出模型不够，只做漂亮页面也不够。真正可交付的 AI 应用，需要能被重新启动、能解释输入边界、能保存证据、能处理失败状态，并且能在训练服务器关闭后继续运行。
